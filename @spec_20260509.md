# @spec_20260509.md - Step 1 FastAPI RSS Monitoring System

## 1. Overview

**Step 1 Deliverable:** A FastAPI-based RSS monitoring system that continuously polls DART and FSC RSS feeds, normalizes events to a standard model, matches against keywords, and routes critical alerts via email while maintaining comprehensive file-based logging.

**Scope:** Automated event collection → normalization → deduplication → keyword matching → severity assignment → notification routing

**Reference:** See detailed architecture rationale in `ideation/ideation_step1_webcrawl_20260509.md`

---

## 2. Event (What triggers the system?)

### 2.1 Startup Event
- **Trigger:** Application server starts (command: `uvicorn main:app --reload`)
- **Action:** Load configurations from YAML files → validate sources → initialize scheduler → start background polling jobs
- **Dependency:** .env file present, config/ directory initialized

### 2.2 Scheduler Trigger (Polling Cycle)
- **Trigger:** APScheduler interval job fires (e.g., every 5 minutes for DART, 10 minutes for FSC)
- **Configuration:** Defined in `sources.yaml` with per-source `poll_interval_seconds`
- **Behavior:** 
  - ThreadPoolExecutor spawns N workers (default 5)
  - Each worker fetches RSS from configured URL
  - All workers complete or timeout (15 seconds per URL)
  - Results aggregated → sent to normalizer

### 2.3 Manual Trigger (API Call)
- **Endpoint:** `POST /trigger?source=dart` or `POST /trigger?source=fsc`
- **Behavior:** Immediate polling of specified source (not waiting for scheduler)
- **Response:** HTTP 202 Accepted (job queued)

### 2.4 System Restart Event
- **Trigger:** Application restarts after graceful shutdown or crash
- **Action:** 
  - Load `data/events.jsonl` (last known events)
  - Rebuild deduplication index from content_hash
  - Identify `failed_alerts.jsonl` and queue for retry
  - Resume scheduler

## 3. Action (What should the system do?)

### 3.1 RSS Collection Flow

**Step 3.1.1: Source Discovery**
- Read `sources.yaml`
- Filter active sources (status: enabled)
- Example structure:
  ```yaml
  sources:
    dart:
      url: "https://dart.fss.or.kr/rss/company"
      poll_interval_seconds: 300
      enabled: true
      timeout_seconds: 10
    fsc:
      url: "https://www.fsc.go.kr/rss/press"
      poll_interval_seconds: 600
      enabled: true
      timeout_seconds: 10
  ```

**Step 3.1.2: HTTP Fetch**
- For each active source:
  - Use `requests.get(url, timeout=timeout_seconds, headers={"User-Agent": "..."})`
  - Retry up to 3 times on network error (exponential backoff: 1s, 2s, 4s)
  - Set timeout to prevent hanging threads
  - Log request: timestamp, URL, response status, duration

**Step 3.1.3: RSS Parsing**
- Use `feedparser` library to parse XML/Atom feeds
- Extract:
  - `external_id` (from entry.id or entry.link)
  - `title` (from entry.title)
  - `url` (from entry.link)
  - `published_at` (from entry.published_parsed → ISO 8601)
  - `summary` (from entry.summary, optional)
  - `raw_payload` (entire entry dict for archiving)
- Handle parsing errors gracefully (log, skip entry, continue)

**Step 3.1.4: Normalization**
- Convert each parsed entry to `ExternalEvent` Pydantic model:
  ```python
  class ExternalEvent(BaseModel):
      source: str  # "dart", "fsc"
      external_id: str
      title: str
      url: str
      published_at: datetime
      fetched_at: datetime
      summary: Optional[str] = None
      raw_payload: dict
      content_hash: str  # MD5(title + url)
      severity: str  # "urgent", "watch", "info" (assigned later)
      matched_keywords: list[str] = []
  ```
- Compute `content_hash = MD5(f"{title}{url}".encode()).hexdigest()`
- Set `fetched_at = now (UTC)`
- Convert `published_at` to UTC ISO 8601

**Step 3.1.5: Deduplication**
- Check if `content_hash` exists in `data/events.jsonl` (bloom filter or simple set for Step 1)
- If exists: skip this event, log "Duplicate detected: {title}"
- If new: proceed to keyword matching
- Dedup index: load from disk at startup, refresh hourly or on-demand

---

### 3.2 Processing Flow

**Step 3.2.1: Keyword Matching**
- Load `keywords.yaml` at startup:
  ```yaml
  keywords:
    urgent:
      - "구조조정"
      - "폐지"
      - "청산"
    watch:
      - "인수합병"
      - "자본증가"
    info:
      - "공시"
      - "보도자료"
  ```
- For each new event, compile keywords into pre-compiled regex patterns (case-insensitive, word boundary `\b`)
- Search title + summary against all patterns
- Return `(highest_severity, matched_keywords_list)`

**Step 3.2.2: Severity Assignment**
- Hierarchy: urgent > watch > info
- Example:
  - Title contains "구조조정" → severity = urgent
  - Title contains "인수합병" → severity = watch
  - No keyword match → severity = info
- Ambiguity handling (Step 1 conservative approach):
  - If multiple severities match, assign the highest
  - Example: "구조조정 아직 검토 중" → severity = urgent (matches urgent keyword, ignores exclusion)

**Step 3.2.3: Deduplication by Content Hash**
- Re-check content_hash against in-memory set
- If collision: mark as duplicate, skip notification, log event to `data/events.jsonl` with `is_duplicate=true`
- Update dedup index

---

### 3.3 Notification Flow

**Step 3.3.1: Routing Decision**
```
if severity in ["urgent", "watch"]:
  → spawn thread for email_send()
  → always write to alerts.jsonl
else:  # severity = "info"
  → write to alerts.jsonl only
```

**Step 3.3.2: Email Notification (Async Thread)**
- Triggered for urgent/watch events
- Configuration from `.env`:
  ```env
  ALERT_EMAIL=recipient@example.com
  SMTP_SERVER=smtp.gmail.com
  SMTP_PORT=587
  SMTP_USER=alerts@company.com
  SMTP_PASSWORD=xxxxx
  ```
- Email send logic:
  - Build MIMEMultipart message
  - Subject: `[URGENT] Event title (max 100 chars truncated)`
  - Body: Source, published_at, url, summary, matched_keywords
  - Retry up to 3 times (exponential backoff: 1s, 2s, 4s)
  - Timeout: 10 seconds per attempt
  - **Failure handling:** If all retries fail, catch exception, log error, write to `data/failed_alerts.jsonl`, **continue processing**
  - Rate limit: max 1 email per second (throttle queue)

**Step 3.3.3: File Logging (Always)**
- Write event to `data/alerts.jsonl` regardless of email success/failure
- Format: One JSON per line (JSONL)
  ```json
  {
    "source": "dart",
    "external_id": "123456",
    "title": "...",
    "url": "...",
    "published_at": "2026-05-09T10:30:00Z",
    "fetched_at": "2026-05-09T10:35:00Z",
    "severity": "urgent",
    "matched_keywords": ["구조조정"],
    "email_sent": true,
    "email_sent_at": "2026-05-09T10:35:05Z",
    "email_error": null
  }
  ```
- Atomic append using lock (threading.Lock)

**Step 3.3.4: Failed Alert Retry (Optional in Step 1, Required in Step 2)**
- If email fails, append to `data/failed_alerts.jsonl` with `retry_count=1, next_retry_at=now+30min`
- Background job every 30 minutes:
  - Read failed_alerts.jsonl
  - For each entry where `next_retry_at <= now`:
    - Attempt email send
    - If success: move to alerts.jsonl, remove from failed_alerts.jsonl
    - If fail: increment retry_count, set next_retry_at
    - If retry_count > 5: log permanent failure, keep in failed_alerts.jsonl

## 4. Requirements

### 4.1 Functional Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| **FR-1** | Poll RSS feeds every N seconds (configurable per source) | DART: 300s, FSC: 600s (configurable in sources.yaml) |
| **FR-2** | Detect new events within 5 minutes of publication | Latency from published_at to detected_at < 5 min |
| **FR-3** | Normalize events to ExternalEvent model | All fields mapped, no data loss, validation passes |
| **FR-4** | Deduplicate events using content_hash | Duplicate events skipped, logged to events.jsonl |
| **FR-5** | Match keywords and assign severity | >95% accuracy on keyword matching (test set) |
| **FR-6** | Send emails for urgent/watch events | Email delivered within 10 seconds of detection |
| **FR-7** | Log all events to alerts.jsonl | 100% of events logged, JSONL format valid |
| **FR-8** | Provide GET /status endpoint | Response includes event_count, alert_count, last_poll, source health |
| **FR-9** | Provide GET /events endpoint (query recent events) | Limit: 1-1000, default 100; returns JSONL |
| **FR-10** | Provide GET /alerts endpoint (query recent alerts) | Limit: 1-1000, default 100; returns JSONL |
| **FR-11** | Handle SMTP server unavailable gracefully | System continues, logs error, queues retry |
| **FR-12** | Detect and recover from corrupted JSONL files | Remove partial lines, preserve valid data, log corruption event |

### 4.2 Non-Functional Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| **NFR-1** | Response time for GET /status | < 200 ms (p99) |
| **NFR-2** | Response time for GET /events (100 items) | < 500 ms (p99) |
| **NFR-3** | JSONL write latency per event | < 50 ms (average) |
| **NFR-4** | Email send timeout | 10 seconds per attempt |
| **NFR-5** | System uptime | 99% (with auto-restart) |
| **NFR-6** | Memory usage under normal load | < 500 MB (100 events in memory) |
| **NFR-7** | Throughput | 100+ events/hour (per source) |
| **NFR-8** | CPU usage | < 5% average (5 worker threads idle) |
| **NFR-9** | Disk usage for 1 month of events | < 500 MB (estimated 50 events/day × 30 days × ~1KB per event) |
| **NFR-10** | Thread pool size | 5 workers (configurable, max 32) |

### 4.3 Constraints

- **No database:** File-based storage only (JSONL, YAML)
- **Python version:** 3.9+
- **Framework:** FastAPI + APScheduler
- **Persistence:** JSONL for events/alerts, YAML for config, .env for secrets
- **SMTP:** Optional configuration (system continues if unavailable)
- **No external infrastructure:** No Celery, Redis, or other message brokers for Step 1
- **File I/O:** Synchronous (threading.Lock for concurrency control)

### 4.4 Security Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| **SEC-1** | API keys in .env only | No hardcoded secrets in config YAML or Python files |
| **SEC-2** | SMTP password masked in logs | All logs sanitize password, only show "****" |
| **SEC-3** | No sensitive data logged | Event titles/URLs allowed, but no SMTP credentials |
| **SEC-4** | JSONL files readable only by app | File permissions: 0600 (read/write by owner only) |
| **SEC-5** | Input validation | All user inputs sanitized (SQL injection N/A, but command injection via keywords) |
| **SEC-6** | Rate limiting (optional) | Max 1 email/sec per destination (step 2 add full rate limiting) |

## 5. Scenario (Happy & Exception Paths)

### 5.1 Happy Path: Normal RSS Collection and Notification

**Setup:**
- System started 1 hour ago
- RSS sources configured (DART: 5min, FSC: 10min)
- keywords.yaml loaded (50 keywords)
- SMTP configured with valid credentials

**Sequence:**
```
T+300s: Scheduler triggers DART poll
  ↓
Fetch https://dart.fss.or.kr/rss/company → HTTP 200, 15 entries
  ↓
Parse → normalize → dedup check
  ↓
13 new events (2 duplicates skipped)
  ↓
Match keywords:
  - 3 events: severity=urgent (구조조정)
  - 4 events: severity=watch (인수합병)
  - 6 events: severity=info (no match)
  ↓
Send emails (3 threads, async):
  - mail.send() for urgent events → success within 2s
  ↓
Write to data/alerts.jsonl (atomic append, lock):
  - 13 records appended
  ↓
Update /status:
  - event_count: 245
  - alert_count: 87
  - last_poll: 2026-05-09T12:05:00Z
  - dart.status: "ok"
```

**Verification:**
- GET /status → 200, all fields populated
- GET /events?limit=13 → 200, JSONL response with new events
- GET /alerts → 200, shows 3 urgent + 4 watch recent
- tail data/alerts.jsonl → 13 new lines appended
- Email inbox → 3 emails received

### 5.2 Exceptional Cases

#### **Case A: SMTP Server Not Configured**

**Precondition:** `SMTP_SERVER` env var missing or empty

**Sequence:**
```
Event detected: severity=urgent
  ↓
EmailNotifier.send_alert()
  ↓
Exception: smtplib.SMTPException("SMTP not configured")
  ↓
Catch → log warning "Email disabled, SMTP not configured"
  ↓
Skip email, continue to file logging
  ↓
Write to data/alerts.jsonl anyway
  ↓
Set email_sent=false, email_error="SMTP not configured"
```

**Verification:**
- Event logged to alerts.jsonl with `email_sent=false`
- System continues, no crash
- No email sent, no retry queue
- Log shows warning: "Email disabled, SMTP not configured"

---

#### **Case B: RSS URL Returns 404 (Source Unavailable)**

**Precondition:** DART RSS URL changed or service down

**Sequence:**
```
Scheduler triggers DART poll
  ↓
fetch_rss_feed(dart_url)
  ↓
requests.get() → HTTP 404, raises RequestException
  ↓
Catch in worker thread, log error:
  "RSS fetch failed: dart (status 404)"
  ↓
Increment source error counter
  ↓
Return empty list []
  ↓
Worker completes (no events from DART)
  ↓
FSC poll continues normally
  ↓
GET /status → dart.status: "error", last_error: "HTTP 404"
```

**Verification:**
- GET /status shows `dart.status="error"`, `dart.last_error="HTTP 404"`
- No crash, FSC events still collected
- Error logged with timestamp
- Alert email NOT sent (no new events)

---

#### **Case C: Duplicate Event Detection**

**Precondition:** Same event fetched twice (e.g., from RSS cache)

**Event 1 at T+300s:**
- Title: "한신금융그룹 구조조정 발표"
- URL: "https://dart.fss.or.kr/123456"
- content_hash: "a1b2c3d4e5f6"
- Logged to events.jsonl
- Email sent (urgent)

**Event 2 at T+600s (duplicate from RSS cache):**
```
Parse entry → same title, same URL
  ↓
Compute content_hash → "a1b2c3d4e5f6" (same)
  ↓
Check dedup index: content_hash in index?
  ↓
YES → log "Duplicate detected: 한신금융그룹..."
  ↓
Skip keyword matching, skip email
  ↓
DO NOT write to alerts.jsonl (or mark is_duplicate=true)
```

**Verification:**
- Event 2 logged to events.jsonl with `is_duplicate=true`
- No email sent for Event 2
- Alert count unchanged
- Log: "Duplicate event skipped"

---

#### **Case D: JSONL File Corrupted (Partial Line After Crash)**

**Precondition:** Process crashed during JSONL write, leaving partial JSON line

**File state:**
```
{"source":"dart","external_id":"1",...}
{"source":"dart","external_id":"2",...}
{"source":"dart","external_id":"3"... (INCOMPLETE, missing closing brace)
```

**On restart:**
```
Load data/alerts.jsonl
  ↓
Read line by line:
  - Line 1: JSON parse OK
  - Line 2: JSON parse OK
  - Line 3: JSONDecodeError (unterminated string)
  ↓
Call validate_jsonl_file()
  ↓
Remove line 3 (last incomplete line)
  ↓
Rewrite file with valid lines only
  ↓
Log warning: "Removed 1 corrupted line from alerts.jsonl"
  ↓
Resume normal operation
```

**Verification:**
- File recovered, valid lines preserved
- Line count reduced (2 valid lines remain)
- Warning logged with line number
- No data loss for lines 1-2

---

#### **Case E: Email Send Timeout (SMTP Server Slow)**

**Precondition:** SMTP server responding slowly (>10s)

**Sequence:**
```
Event: severity=urgent, matched_keywords=["구조조정"]
  ↓
spawn EmailNotifier thread
  ↓
smtplib.SMTP(timeout=10)
  ↓
Connection slow, 15 seconds elapsed
  ↓
socket.timeout exception raised
  ↓
Catch → retry #1 (backoff 1s)
  ↓
Attempt #2 → socket.timeout again
  ↓
retry #2 (backoff 2s)
  ↓
Attempt #3 → socket.timeout again
  ↓
All retries exhausted
  ↓
Exception caught, write to data/failed_alerts.jsonl:
  {
    "event_id": "...",
    "retry_count": 1,
    "next_retry_at": "2026-05-09T13:05:00Z",
    "error": "socket.timeout"
  }
  ↓
Write event to alerts.jsonl with email_sent=false
  ↓
Log error: "Email send failed after 3 attempts, queued for retry"
```

**Verification:**
- Event saved to alerts.jsonl (not lost)
- Failed attempt logged to failed_alerts.jsonl
- Email not sent (retry scheduled for +30min)
- GET /status shows alert_count + failed_count
- No system crash

### 5.3 Edge Cases

#### **Edge A: Burst of 1000 Events in 1 Minute**

**Precondition:** RSS feed returns 1000 entries in single poll (unusual but possible)

**Sequence:**
```
fetch_rss_feed(url) → 1000 entries
  ↓
Normalize all 1000 → ExternalEvent objects
  ↓
Dedup check: Pydantic validation + hash computation
  ↓
1000 objects in memory (~500KB)
  ↓
Keyword matching: iterate 1000 × 50 keywords
  ↓
Match results: ~200 urgent, ~300 watch, ~500 info
  ↓
Batch write to alerts.jsonl:
  JSONL.append_batch(records) with lock
  ↓
Email spawn 200 threads (throttle: 1/sec)
  ↓
Email queue builds up (198 pending)
  ↓
Thread pool eventually completes all emails
  ↓
GET /status: event_count + 1000, no crashes
```

**Verification:**
- Memory peak: <1 GB (within limit)
- JSONL write: batch append, single lock acquisition
- Email queue: doesn't block main loop
- System responsive (GET /status within 200ms)
- All 1000 events logged eventually

---

#### **Edge B: System Crash During JSONL Write**

**Precondition:** Process killed during `f.write(json.dumps(record) + '\n')`

**Before crash:** File pointer at 1000 bytes
**Kill signal:** SIGTERM
**After restart:**

```
On startup:
  ↓
Load events.jsonl for dedup index
  ↓
Read line by line, validate JSON
  ↓
Detect corruption at position (incomplete last line)
  ↓
Call validate_jsonl_file()
  ↓
Remove corrupted line, rewrite file
  ↓
Rebuild dedup index from valid lines
  ↓
Resume scheduler
```

**Verification:**
- File integrity restored
- Dedup index rebuilt from valid data
- No duplicate events on restart (re-poll is idempotent)
- Log: "Recovered from 1 corrupted line"

---

#### **Edge C: Timezone and Timestamp Handling**

**Precondition:** RSS feeds mix UTC and local time (DART in UTC, FSC in Asia/Seoul)

**DART entry:** `<pubDate>Thu, 09 May 2026 10:30:00 GMT</pubDate>`
**FSC entry:** `<pubDate>2026-05-09T10:30:00+09:00</pubDate>` (Korea time)

**Normalization:**
```
Parse DART:
  → feedparser converts to UTC tuple
  → published_at = "2026-05-09T10:30:00Z" (UTC)

Parse FSC:
  → feedparser handles +09:00 offset
  → Normalize to UTC: 2026-05-09T10:30:00+09:00 → 2026-05-09T01:30:00Z
  → published_at = "2026-05-09T01:30:00Z" (UTC)
```

**Verification:**
- All timestamps stored as ISO 8601 UTC
- Comparison/sorting works correctly
- API responses show UTC (with Z suffix)
- No timezone conversion errors

---

#### **Edge D: Very Long Event Titles (>500 characters)**

**Precondition:** RSS entry with unusually long title (e.g., summary embedded)

**Title:** "신한금융그룹 2026년 상반기 사업전략 발표 회의록 (2026-05-09) - 장황한 설명이 포함된 매우 긴 제목 블라블라 ..."  (500+ chars)

**Email subject:** Field has RFC 5322 limit (~100 chars recommended)

**Sequence:**
```
Event normalized:
  title = "...긴제목..."  (500 chars)
  ↓
Email construction:
  subject = f"[URGENT] {event.title}"
  ↓
Truncate subject to 100 chars:
  subject = "[URGENT] 신한금융그룹 2026년 상반기 사업전략 발표 회의록 (2026-05-09) - 장황한..."
  ↓
Body: full title preserved
  ↓
alerts.jsonl: full title stored
  ↓
Log: "Title truncated from 500 to 100 chars (subject line)"
```

**Verification:**
- Email sent with truncated subject
- Full title preserved in JSONL
- No validation errors
- Log shows truncation

## 6. API Contract

### GET /status

**Description:** System health and event statistics

**Response:**
```json
{
  "status": "healthy",
  "uptime_seconds": 3600,
  "last_poll": "2026-05-09T12:05:00Z",
  "event_count": 245,
  "alert_count": 87,
  "failed_alert_count": 2,
  "sources": {
    "dart": {
      "last_poll": "2026-05-09T12:05:00Z",
      "status": "ok",
      "event_count": 145,
      "alert_count": 45,
      "error_count": 0
    },
    "fsc": {
      "last_poll": "2026-05-09T12:00:00Z",
      "status": "ok",
      "event_count": 100,
      "alert_count": 42,
      "error_count": 0
    }
  },
  "memory_mb": 245,
  "cpu_percent": 2.5
}
```

**Status codes:** 200 OK

---

### GET /events

**Description:** Query recent events (all events, regardless of severity)

**Query parameters:**
- `limit` (integer, 1-1000, default 100): number of recent events to return
- `source` (string, optional): filter by "dart" or "fsc"
- `days` (integer, optional): filter last N days

**Response:** JSONL format (one JSON object per line)

```
{"source":"dart","external_id":"123","title":"...","published_at":"2026-05-09T10:30:00Z",...}
{"source":"fsc","external_id":"456","title":"...","published_at":"2026-05-09T10:25:00Z",...}
```

**Status codes:** 200 OK, 400 Bad Request (invalid limit/source)

---

### GET /alerts

**Description:** Query recent alerts (events with severity = urgent or watch)

**Query parameters:**
- `limit` (integer, 1-1000, default 100)
- `severity` (string, optional): filter by "urgent", "watch", or "info"
- `days` (integer, optional)

**Response:** JSONL format

**Status codes:** 200 OK, 400 Bad Request

---

### POST /trigger

**Description:** Manually trigger polling for a specific source

**Request body:**
```json
{
  "source": "dart"
}
```

**Response:**
```json
{
  "job_id": "abc123",
  "source": "dart",
  "status": "queued",
  "message": "Poll job queued"
}
```

**Status codes:** 202 Accepted, 400 Bad Request (unknown source), 503 Service Unavailable (scheduler offline)
