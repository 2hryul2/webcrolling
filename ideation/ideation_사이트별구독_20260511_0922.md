# Ideation: 사이드바 사이트별 체크박스 구독 기능

**작성일**: 2026-05-11
**대상 feature**: 카테고리별로 소속 사이트를 펼쳐 보여주고, 사이트마다 체크박스로 모니터링 ON/OFF
**작성 단계**: Ideation (Phase 1) — 본 문서 통과 후 Spec(EARS) → 구현 순.
**현재 baseline**: Step 4 deployed (130 tests pass, commit `f847771`)

## 1. Context (왜 이 기능을 추가하는가)

**현재 UI** ([static/watchtower.html](..\static\watchtower.html) 538–778)
- 사이드바 "내 구독 카테고리" / "기타 카테고리" — **카테고리 8개**만 노출.
- 각 행: ⭐ 별(카테고리 구독) + 🔔 벨(채널: instant/digest/off) + 미확인 카운트.
- 사이트(30개)는 사이드바에 없음. 메인 피드 카드의 출처 라벨로만 노출.

**요구사항**: 사이드바 내 구독 카테고리 영역에서 **카테고리별로 소속 사이트 목록을 펼쳐 보여주고**, **사이트마다 체크박스로 모니터링 ON/OFF** 토글.

**의도/효용**
- 한 카테고리 안에서도 일부 사이트(노이즈/저신뢰)만 끌 수 있게.
- 사이트 단위 가시성 확보 — "이 시스템이 무엇을 보고 있는지" 사용자가 직접 점검.
- 향후 사이트 등록 API(Step 5 FR-SITE-002) 도입 시에도 같은 UX로 확장 가능.

## 3. 접근법 비교 (3가지)

### A. Mute 오버라이드 (sparse negative)
- 신규 모델: `SiteSubscription(user_id, site_id, muted: bool)` — `muted=True` row만 존재.
- Default 의미: "카테고리 subscribed=True → 그 안의 모든 사이트 자동 포함, 단 muted row만 제외".
- 알림 필터: `... AND NOT EXISTS (SELECT 1 FROM site_subscriptions WHERE muted=True AND site_id=Item.site_id)`
- 장점: 신규 사이트 자동 포함, DB row 적음.
- 단점: **negative 모델** — UX 직관 떨어짐, audit/디버깅 row 부재 case가 까다로움.

### B. Watchlist (positive whitelist)
- 신규 모델: `SiteSubscription(user_id, site_id, enabled: bool)` — 모든 (user, site) 쌍에 1행 dense.
- Default 의미: 시드 시 30 사이트 × 1 사용자 = 30행 `enabled=False`.
- 알림 필터: `JOIN site_subscriptions ON site_id WHERE enabled=True`.
- 장점: positive 모델, audit 우수, 신규 사이트 시 명시적 opt-in.
- 단점: 새 사이트 추가 시 기본값 결정 정책 필요.

### C. Subscription 재정의 (user × site로 통합)
- 신규: `Subscription(user_id, site_id, subscribed, channel)` — 카테고리 별/벨은 그루핑 표시용으로만.
- 장점: 사이트 granularity 본질적, 모델 일관.
- 단점: Step 4 자산 대거 재작성 (FR-SUB-001~005, FR-NOTIF, 23 테스트). 카테고리 단위 다이제스트 의미 약화.

## 4. 사용자 확정 결정 (2026-05-11)

| 결정 항목 | 채택 | 비고 |
|---|---|---|
| 모델 접근법 | **C 재정의** | 단 카테고리 별/벨 기존 의미 유지를 동시 채택 → 두 테이블 공존 형태로 정합화 (§5) |
| 카테고리 ⭐ 별 의미 | **기존 유지 + AND 조건** | 카테고리 별 ON · 채널 ∈{instant,digest} AND 사이트 enabled 둘 다 True일 때만 알림 |
| 신규 사이트 default SiteSubscription | **카테고리 별 ⭐ 상태 따름** | 사이트 등록 시점에 사용자가 그 카테고리 별 ON이면 자동 `enabled=True`, 아니면 `False` |
| Accordion 펼침 상태 영속 | **제외** | 매 로드 시 "내 구독" expand / "기타" collapse 기본. localStorage 제거 정책 유지 |
| 사이트별 채널 분기 | **제외** | 채널(instant/digest)은 카테고리 단위로만. 사이트는 ON/OFF만 |
| 사이트 상태 배지 | **표시** | `Site.status` ∈ {ok,delayed,failed,blocked} 작은 색 점 |

### 4-1. C + "카테고리 별 유지" 정합화

> C("Subscription 재정의")와 "카테고리 별 기존 유지 + AND 조건"을 합치면 **양쪽 테이블이 공존하면서 site 레벨이 UX/spec의 일차 surface**가 되는 형태가 가장 자연스럽다:
>
> - 기존 `Subscription` 테이블은 그대로 두되 **이름을 `CategorySubscription`으로 리네이밍**하여 의미 명확화 (이번 사이클 마이그)
> - 신규 `SiteSubscription(user_id, site_id, enabled, updated_at)` 테이블 추가 — 사이트 단위가 진실의 주축
> - 알림 필터는 두 테이블의 **AND 조건**
> - Spec/UI 설명문은 site 단위를 일차로 서술, 카테고리는 "관심 영역 게이트 + 알림 채널" 보조 surface

## 5. 최종 데이터 모델

### 5-1. 리네이밍 (마이그)

| Before | After | 변경 사유 |
|---|---|---|
| `subscriptions` (user, category, subscribed, channel) | `category_subscriptions` (동일 컬럼) | "구독" 용어를 site 레벨이 차지 → 카테고리 행위 명확화 |

> **마이그 전략**: SQLAlchemy 클래스 리네이밍만으로는 SQLite 기존 테이블 자동 이전 안 됨. 1회성 함수 `migrate_subscriptions_to_category_subscriptions(session)` 신설 — 부팅 시 lifespan에서 옛 테이블이 존재하면 새 테이블로 SELECT→INSERT 후 옛 테이블 DROP, 트랜잭션 처리. Alembic 미사용 정책은 Step 5 의제 그대로.

### 5-2. 신규 테이블
```python
class SiteSubscription(Base):
    __tablename__ = "site_subscriptions"
    __table_args__ = (UniqueConstraint("user_id", "site_id", name="uq_user_site"),)
    id: str = uuid hex
    user_id: FK users
    site_id: FK sites
    enabled: bool default=False
    updated_at: datetime default=now, onupdate=now
```

- 시드 `_seed_site_subscriptions(session)` — 1×30=30행 멱등 적재, 초기 `enabled=False`.

### 5-3. 알림 필터 (AND 조건)
```sql
-- send_instant
SELECT User.* FROM users U
  JOIN category_subscriptions CS ON CS.user_id = U.id
  JOIN site_subscriptions     SS ON SS.user_id = U.id
WHERE CS.category_id = :cat_id
  AND CS.subscribed  = TRUE
  AND CS.channel     = 'instant'
  AND SS.site_id     = :site_id
  AND SS.enabled     = TRUE
```
- digest 쿼리도 동일 패턴 (`channel='digest'`).
