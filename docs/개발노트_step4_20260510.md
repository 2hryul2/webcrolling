## 1. 목표

Step 3에서 적재된 items를 사용자가 카테고리별로 구독·관리하고, 즉시·다이제스트·5회 실패 메일을 SMTP로 발송하며, UI 상태를 localStorage에서 서버 영속으로 전환한다.

**Spec FR 매핑:**
- FR-SUB-001~005 — 구독 모델 + 권한 분리 + 자동 전환 (별 OFF → 채널 'off', 채널 instant/digest → 별 ON)
- FR-NOTIF-001~008 — 60초 이내 즉시 / 09:00 KST 다이제스트 / 60·300·900s 백오프 / 5분·10건 rate limit / 읽음 제외
- FR-FEED-006 — 읽음 처리 영속
- FR-SITE-006 — 5회 실패 시 owner 메일 알림

**Out of Scope (Phase 1):**
- CHANGE 감지 (Phase 2)
- HTML snapshot + diff (Phase 2)
- 사내 메신저 webhook (Phase 2)
- 사내 SSO + 토큰 인증 (Step 5)
- Audit log + Docker (Step 5)

## 4. 핵심 학습 / 회고

### 4.1 잘된 점
- **Brief의 메일 템플릿 직접 등재** — Builder가 한국어 문구 추측 없이 명시값 그대로 사용. 검토 시점에 디자인 논의 0건.
- **Step 1 notifier 분리 결정** — 추상화 시도 시 keyword severity ↔ category subscription 의미 충돌. 명확한 모듈 분리가 향후 Step 5 재구조화 시에도 비용 적음.
- **rate limit in-memory 채택** — DB 테이블 신설 회피 → schema 단순. Phase 1 단일 프로세스 ASM과 정합.
- **localStorage 완전 제거** — fallback 유지 시 "어느 상태가 진실인지" 분기. 단일 진실 모델이 디버깅·테스트 비용 적음.

### 4.2 아쉬운 점
- **selector 보정 (s10/s12/s16/s19/s24/s29)** — Step 3에서 이월된 작업, Step 4에서도 외부 환경 제약으로 미완. Step 5에서도 동일 패턴이면 Owner 환경 의존이 영구 차단 요인. 별도 검증 환경(폐쇄망 사외 게이트웨이) 마련 검토 필요.
- **rate limit 5분/10건 단순화** — spec FR-NOTIF-007의 "11번째부터 묶음" 정확 의미는 "한 번의 다이제스트로 묶여 발송"인지 "다음 09:00 다이제스트에 포함"인지 모호. Step 4에서는 "skip 후 다음 09:00 digest에 포함" 으로 단순 해석. spec v0.2에서 명시 필요.
- **AlertLog.detail 컬럼 활용 미검증** — 다이제스트 row의 `detail='N items in M categories'` 포맷은 brief 명시값이지만 실 사용 (운영 대시보드 등) 시점에 재검토 필요할 수 있음.

## 6. 운영 노트 (Step 4 deployed 후)

- **SMTP 미설정 시**: 부팅 정상, AlertLog 에 status='skipped' 행만 기록. 워커는 계속 진행. 즉시 알림이 필요한 운영 환경 진입 전 `.env`에 SMTP_*+ALERT_EMAIL 필수.
- **다이제스트 첫 발송**: 서버 24h 이상 운영 + 다음 KST 09:00 도달 시. 테스트하려면 `notifier_service.send_digest()` 직접 호출 또는 cron trigger 시간 변경.
- **rate limit 카운터**: 프로세스 재기동 시 리셋. uvicorn 재시작 ≠ 누적 보존. 운영 단일 프로세스 가정 위반 시(예: `--workers 4`) 카운터가 워커별로 분리됨 — Phase 2 외부 캐시(Redis) 검토.
- **owner failure 메일**: `_failure_notified` set이 1회 알림 보장. 사이트 복구(연속 성공) 시 카운터 리셋 + set 제거 → 다시 5회 실패 시 알림 재발송.
- **localStorage 잔재 제거**: `static/watchtower.html`의 `STORAGE_KEY`/`loadState`/`saveState`/`DEFAULT_STATE` 모두 삭제됨. 사용자 브라우저에 옛 키가 남아있을 수 있으나 코드 미참조 → 무해.
